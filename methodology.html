What this is

Ocean Intel is a daily-refreshed, audit-grade index of distress signals attached to Ocean County properties. The goal is the operator's goal: surface the relatively small set of parcels where multiple distinct distress patterns have stacked on the same property. That stacking is far more predictive than any single signal, and far more honest than a black-box "motivated seller score."

The pipeline is one-way: scrapers fetch raw public records, the build step joins and patterns them into data/leads.json, the dashboard renders that file. There is no database, no ML model, no proprietary scoring weights. Every count, every tier, every pattern membership is reproducible from the raw input files.

Daily refresh is the product. Vendors like PropStream and DealMachine buy bulk monthly extracts. Ocean Intel hits source endpoints directly, so a sheriff sale recorded Tuesday morning is on the dashboard Wednesday morning. The freshness gap is the moat.

The 6-pattern stack

Every distress signal we collect is mapped to one (or two) of six orthogonal patterns. Orthogonal meaning the patterns don't overlap — a property in foreclosure isn't also in tax distress unless a separate tax-derived signal also fires.

PatternMeaningNJ-specific signals that fire it
jfcJudicial foreclosureSheriff PDF entries (active or adjourned). Lis pendens, notices of pending sale (clerk records — pending unblock).
taxTax distressTax sale certificates, federal/state tax liens, in-rem municipal tax foreclosures (clerk records — pending unblock). NJ MOD-IV tax-delta heuristics (when LAST_YR_TX gaps are unusual).
estateProbate / decedentNJ inheritance tax waivers, disclaimers (heir refusing inheritance), trust agreements, executor/administrator deeds (clerk records — pending unblock). Surrogate dockets are not exposed publicly.
codeMunicipal code enforcementSparse — code is municipal in NJ, rarely county-recorded. Fires only when a clerk-recorded municipal lien (a subset of INSTLIEN) hits the official records.
lienNon-tax encumbrancesNJ construction liens (the legal name for mechanic's liens), physician's liens, hospital liens, judgment liens, writs of execution.
transferDistressed transferQuitclaim deeds, $1 / $0 nominal-consideration deeds, sheriff's deeds, executor's/administrator's deeds. The MOD-IV layer alone fires this whenever a recent deed price ≤ $10.

How a property becomes Hot, Warm, or Active

Tier comes from the count of distinct patterns that fire on a property — not from a sum of weighted scores.

n = number of distinct patterns that fired on this parcel
if n >= 3:  tier = "hot"
elif n == 2: tier = "warm"
elif n == 1: tier = "active"
else: # n = 0 — drop, not a lead

This is deliberate. The framework spec calls out "score inflation" as a top failure mode — stack four reasonable bonuses on top of each other and your top-decile threshold becomes meaningless. Counting distinct patterns flips that: a property with three different kinds of distress (foreclosure + tax lien + decedent transfer, say) is qualitatively different from a property with three filings of the same pattern.

Each pattern caps at 3 most-recent signals so the JSON file size stays under GitHub's 50 MB hard cap, and so the freshness signal isn't drowned out by historical clutter.

The Two-Truths invariant

The dashboard's stat tiles ("3 Hot, 17 Warm, 482 Active") and the rendered table rows are derived from the same matches(lead) function. There's no separate counter. There's no second filter pipeline. The single source of truth is enforced two ways:

  1. In pipeline/build_leads.py — before leads.json is written, the header's tier_counts and pattern_counts are recomputed by walking the records list. If they disagree, the pipeline raises and exits non-zero. leads.json is never written with drifted counts.
  2. In the dashboard — the table body and the stat tiles both call RECORDS.filter(matches). There is no parallel filter logic.

This guards against the most common dashboard failure: filter shows "1,200 results" but the tile says "5,000" because two different code paths computed each.

Doc-type taxonomy

The clerk's publicsearch exposes 168 distinct doc-type abbreviations. A doc-type filter pill on the dashboard surfaces exactly the abbreviations that appear in the current leads.json's signals — not all 168, just the ones we have actually attached. Each abbreviation is independently clickable.

The full pattern map (which doc-type fires which pattern) is in RECON.md on GitHub.

No reCAPTCHA solving. Ocean Intel does not use third-party CAPTCHA-solving services, and does not headless-browser past consent gates. When a source is blocked, we say so. The workaround is OPRA (NJ's Open Public Records Act) for monthly bulk requests, or human-in-loop for ad-hoc seeded sessions.

Known limitations & what's NOT in the data

  • Owner names are redacted upstream. The NJ DOIT public MOD-IV layer returns empty OWNER_NAME fields. We surface mailing address (which may be P.O. Box or out-of-state), situs address, year built, sales history, and full assessment, but the named-owner field is not available without the clerk-search unblock or paid commercial sources.
  • No skip trace. Phone numbers and email addresses are not derived from public records. The CSV export includes empty phone1 / phone2 / phone3 / email1 / email2 columns for downstream skip-trace tooling.
  • No AVM, no ML scoring, no MLS feed. "Equity %" filtering is a planned filter; today the slider is non-functional pending an AVM data source. The framework spec calls out heuristic inflation as a failure mode — we'd rather ship a missing feature than a misleading one.
  • No mortgage balance / payoff data. We see deed book/page and judgment amount (when foreclosure is active), but not current loan balance.
  • Code enforcement is municipal, not county. Until per-municipal feeds are added, the code pattern fires only on the rare clerk-recorded municipal lien.
  • Probate (Surrogate) data is OPRA-only. Until a public docket interface exists, the estate pattern fires only via clerk-recorded estate-deed and inheritance-tax-waiver instruments — which are themselves currently blocked behind the clerk reCAPTCHA wall.
  • Sheriff PDF column-split errors. About 5% of multi-lot sheriff sales have layout-driven parser errors where the zip code or municipality wraps vertically across columns. The block + lot is correctly extracted in nearly all cases, so the parcel join still succeeds; the situs address shown for those rows may be incomplete. We chose to keep the affected rows visible (with a tagged orphan flag where needed) rather than drop them.

Audit trail

Every leads.json the dashboard renders includes a header with:

  • generated_at — UTC timestamp of the build
  • source_commit — short git SHA of the repo at build time
  • tier_counts, pattern_counts, doc_type_counts — Two-Truths-invariant counts
  • source_attach_counts — how many signals each source contributed
  • warm_tier_high_confidence_pct — % of warm leads where both signals are within the last 365 days
  • top_pattern_combos — most common pattern stacks (e.g., [jfc, lien], count)
  • clerk_heartbeat — Ocean Clerk's last-recorded-instrument metadata at refresh time, so day-over-day staleness is detectable

Source code, source commit, and the recon report are all on GitHub: github.com/xcerebroai/ocean-intel.
