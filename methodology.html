What this is

An audit-grade index of distress signals attached to Ocean County properties. The goal is the operator's goal: surface the relatively small set of parcels where multiple distinct distress patterns have stacked on the same property. Stacking is more predictive than any single signal, and more honest than any black-box "motivated seller score."

The pipeline is one-way: scrapers fetch raw public records, the build step joins and patterns them into data/leads.json, the dashboard renders it. No database, no ML, no proprietary scoring. Every count is reproducible from raw inputs. Counts on this page are the actual current numbers — no figures are hand-typed.

Today's snapshot: 298,147 parcels indexed (enrichment lookup), 148 leads after pipeline, 157 total signals, 0 leads new in last 24h. Most-stacked parcel: 2 distinct lead types.

How leads get into this dashboard

The Ocean County Clerk's office records every recorded property instrument — deeds, mortgages, liens, judgments, foreclosure filings. Those records are the leads. Until the operator unblocks the clerk's data feed, this dashboard shows only sheriff foreclosure sales (the one source not behind reCAPTCHA).

Two paths to unblock — both ship in this build, both run automatically once configured.

Path A — OPRA bulk request (system of record)

A pre-filled, signature-ready PDF is generated monthly at opra_requests/<YYYYMM>_ocean_clerk.pdf. The request asks for a machine-readable bulk export (CSV/Excel) of every recorded instrument in the prior calendar month, filtered to the doc-type abbreviations that fire each pattern. Cites N.J.S.A. 47:1A-1, requests electronic delivery (drops the per-request fee to near zero), and asks for standing monthly delivery for 12 months so the operator does not need to re-submit.

  1. Sign the generated PDF.
  2. Email it to the Ocean County Clerk's office (custodian: County Clerk John P. Kelly).
  3. Response within ~7 business days.
  4. When the operator receives the CSVs, drop them into data/raw/clerk_opra/incoming/.
  5. Next refresh runs scrapers/clerk_opra_ingest.py automatically — leads appear in the dashboard. No code changes needed.

Path B — Seeded session (daily freshness layer)

The clerk's NewVision publicsearch API enforces reCAPTCHA v3 server-side. We do not solve CAPTCHA. Instead, the operator opens the clerk site in their real Chrome, completes one search (which triggers the silent reCAPTCHA challenge their session passes), and copies the resulting cookies into .env. The pipeline replays them on automated calls.

  1. Open https://sng.co.ocean.nj.us/publicsearch/ in Chrome.
  2. Submit any doc-type search (e.g. DEED, last 7 days).
  3. DevTools → Application → Cookies → copy ALL cookies for sng.co.ocean.nj.us as a single name=value;name=value;... string.
  4. Paste into .env: 
    CLERK_SESSION_COOKIES=ASP.NET_SessionId=...; ...
CLERK_SESSION_TOKEN=<X-RequestVerificationToken header value>
CLERK_SESSION_SEEDED_AT=2026-05-05T13:42:00Z
  5. Pipeline pulls daily until session expires (typically 24–72h). On expiry, refresh harness sends a Telegram alert: "Re-seed clerk session within 24h."
  6. Re-seed weekly — 30 seconds.

Both paths run in parallel. OPRA is the system of record (monthly bulk, complete coverage). Seeded session is the daily freshness layer (catches new filings within hours of recording). Output schema is identical, so the pipeline doesn't care which path the data came from.

Why MOD-IV parcel data is enrichment, not a lead source

The NJ MOD-IV statewide parcel layer publishes assessed values, sales history, mailing addresses, year built, and ~40 other parcel-state fields for all 298,147 Ocean parcels. This is structural metadata about a property — not a recorded distress event. The v1.0 build mistakenly treated MOD-IV nominal-consideration deeds (sale price ≤ $10) as distressed-transfer leads, generating 6,921 records that were almost entirely family / trust / corporate-restructuring transfers with no investor relevance. That noise drowned out the 80 real sheriff-sale leads.

v2.1 reclassifies MOD-IV correctly: parcel data enriches every clerk-derived and sheriff-derived lead with owner mailing address (drives the absentee + out-of-state attributes), assessed value (drives high-equity), deed date (drives long-term-owned + senior-owner proxy), entity-name regex on the mailing line, and so on. MOD-IV does not generate leads. Clerk records and sheriff sales generate leads; MOD-IV decorates them.

Distressed-transfer subtypes (Quitclaim Deed, Sheriff's Deed, Executor's/Administrator's Deed, Deed in Lieu) are real lead types — but they fire from clerk DEED records with grantor/grantee/consideration/sub-type metadata, not from the MOD-IV parcel layer alone.

Match modes (ANY / ALL / 2+ / 3+)

The filter rail lets you check any combination of lead-type patterns, subtypes, and attributes. The match-mode selector decides how the checks combine:

  • ANY (default) — show leads matching any of the checked filters. Equivalent to OR.
  • ALL — show leads matching every checked filter. Equivalent to AND.
  • 2+ — show leads matching at least 2 of the checked filters.
  • 3+ — show leads matching at least 3 of the checked filters.

TTL rules (signal expiry)

Time-bounded signals expire to keep the dashboard signal-to-noise high. Sheriff sale signals additionally expire if not seen in 4 consecutive refreshes (~4 days for daily refresh). Stale signals are dropped before leads.json is written; leads with no remaining signals are removed.

Two-Truths invariant

The dashboard's stat tiles ("Total: 148, Foreclosure: 148, Tax: 0, …") and the rendered table rows are derived from the same matches(lead) function. There is no separate counter, no parallel filter pipeline. The single source of truth is enforced two ways:

  1. In pipeline/build_leads.py — before leads.json is written, the header's pattern_counts, attribute_counts, lead_type_subtype_counts, stack_depth_distribution, total_signals, and most_stacked_count are recomputed by walking the records list. If they disagree, the pipeline raises and exits non-zero. leads.json is never written with drifted counts.
  2. In the dashboard — the table body, the lead-type stat tiles, and the quality stat tiles all call RECORDS.filter(matches). There is no parallel filter logic.

Known limitations

  • Owner names redacted upstream. NJ DOIT publishes the public MOD-IV layer with empty OWNER_NAME. The dashboard surfaces mailing/situs addresses, year built, sales history, and full assessment values, but the named-owner field is not available without the clerk-search unblock or paid commercial sources.
  • No skip trace. CSV export includes empty phone1-3 / email1-2 columns for downstream skip-trace tooling.
  • No AVM / no MLS / no ML scoring. "High equity" and "Free-and-clear" attributes use proxies (assessed-value vs sale-price ratio for high equity; mortgage absence requires clerk_search). These will improve when external sources are wired.
  • Code pattern is municipal in NJ, not county. Until per-municipal feeds are added (33 municipalities in Ocean County), the code pattern fires only via clerk-recorded municipal liens.
  • Surrogate, civil court, and judgment data sit behind login or Imperva walls. OPRA fallback documented for each.
